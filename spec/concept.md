# Концепт-ТЗ проекта rythm

**Версия:** 0.1 MVP  
**Дата фиксации:** 21 мая 2026
**Тип продукта:** небольшой личный PWA-сайт для распределения времени в неделе  
**Основная идея:** пользователь собирает свой недельный жизненный ритм как “билд” из активностей: сон, работа, обучение, спорт, еда, отдых, бытовые дела и другие занятия. Система помогает увидеть, куда уходят 24 часа в сутках и 168 часов в неделе, чем приходится жертвовать ради новых активностей и как меняется баланс жизни.

---

## 1. Краткое описание продукта

**rythm** — это личный веб-инструмент для планирования повторяющегося жизненного сценария недели.

В отличие от Google Calendar, где фиксируются конкретные события, встречи и разовые планы, rythm описывает не календарь событий, а **шаблон жизненного режима**: как обычно распределяется время в течение недели.

В MVP есть один рабочий сценарий недели без переключения между шаблонами. Пользователь:

1. Создаёт иерархию активностей.
2. Назначает активностям цвета и дефолтную длительность.
3. Возвращается на недельную таблицу.
4. Раскидывает активности по дням.
5. Меняет длительность, начало и конец блоков.
6. Смотрит статистику по дням и за неделю.
7. Может работать офлайн через PWA.
8. При появлении интернета синхронизирует состояние с сервером по принципу “более свежая версия побеждает”.

Ключевая метафора: **закуп характеристик в DnD**, только вместо характеристик распределяются часы жизни.

## 2. Цели MVP

### 2.1. Продуктовые цели

- Дать пользователю быстрый способ увидеть всю неделю целиком.
- Позволить перераспределять время при изменении жизненных обстоятельств.
- Показать, чем пользователь жертвует ради новой активности.
- Отделить повторяющийся жизненный сценарий от событийного календаря.
- Сделать инструмент доступным с ПК и телефона.
- Сделать работу возможной без интернета.
- Избежать сложной инфраструктуры: минимум зависимостей, простой деплой, простое хранение данных.

### 2.2. Технические цели

- Простое PWA-приложение на Svelte без тяжёлой инфраструктуры вокруг фронтенда.
- Фронтенд: Svelte 5 + TypeScript, сборка через Vite, без CDN.
- CSS: локальный Bootstrap 5 и Bootstrap Icons из `public/vendor`, плюс кастомный дизайн поверх них.
- Backend: простой Node.js/Express HTTP-сервер.
- Хранение: JSON-файл как основной вариант для MVP.
- PWA: приложение открывается и редактируется офлайн.
- Docker + docker compose для простого развёртывания на VPS.
- Сервер должен автоматически подниматься после перезапуска VPS или падения контейнера.

## 3. Основная модель продукта

### 3.1. Неделя

Неделя — основная рабочая единица продукта.

В MVP существует только одна неделя-сценарий. Это не конкретная календарная неделя с датами, а повторяющийся недельный ритм.

Неделя должна стремиться к сумме **168 часов**. При этом приложение не должно жёстко запрещать ошибки. Если пользователь создал неправильную структуру, приложение подсвечивает предупреждение, но не удаляет и не исправляет данные насильно.

### 3.2. День

День в rythm — это логический день сценария, а не обязательно строго промежуток с 00:00 до 23:59.

Примеры:

- один день может начинаться в 07:00 и заканчиваться около 23:00;
- другой день может начинаться в 08:00 и уходить за полночь;
- блок сна может попадать на следующий календарный день;
- пользователь может вручную резать сон на части, если ему так удобнее.

Главное требование: **начало и конец недели должны совпадать по часам**. Если неделя начинается в понедельник в 07:00, то после завершения седьмого дня она должна логически прийти к следующему понедельнику в 07:00. Если это не так, приложение показывает предупреждение.

### 3.3. Системная активность “Конец дня”

Для разделения логических дней вводится системная активность:

**Конец дня**

Это не обычная активность и не учитывается в статистике времени. Она нужна для того, чтобы сказать приложению: “после этой точки начинается следующий день”.

Правила:

- “Конец дня” отображается в недельной таблице как тонкий системный разделитель.
- После “Конца дня” следующие блоки попадают в следующую колонку дня.
- Если пользователь создал меньше 7 завершённых дней, приложение показывает предупреждение.
- Если пользователь создал больше 7 завершённых дней, приложение показывает предупреждение, но не удаляет лишние дни.
- Если получилось 8 дней вместо 7, восьмой день можно отрисовать как дополнительную колонку с визуальным статусом ошибки.
- “Конец дня” не считается в статистике.
- “Конец дня” нельзя удалить автоматически без подтверждения пользователя.

## 4. Экраны MVP

В MVP нужно 3 основных экрана:

1. **Неделя** — главная псевдотаблица с днями и активностями.
2. **Активности** — иерархия групп и активностей.
3. **Статистика** — сводка по группам, активностям, дням и неделе.

Дополнительно нужен небольшой экран или модальное окно настроек, но его можно встроить в общий интерфейс.

# 5. Экран 1 — “Неделя”

## 5.1. Назначение экрана

Главный экран приложения. Пользователь должен сразу видеть свою неделю и быстро менять распределение времени.

Экран должен отвечать на вопросы:

- чем занят каждый день;
- сколько длится каждое занятие;
- где есть пустоты;
- где можно вставить новую активность;
- где можно заменить активность;
- не сломана ли структура недели;
- сколько дней получилось в сценарии.

## 5.2. Общий вид

Экран выглядит как узкая псевдотаблица:

- колонки — дни недели;
- строки внутри колонок — блоки активностей;
- каждая активность отображается отдельной карточкой;
- карточки достаточно тонкие, чтобы день помещался на экране максимально компактно;
- ширина колонок небольшая;
- на мобильных устройствах вся недельная таблица скроллится горизонтально.

Рекомендуемые подписи дней:

- Пн
- Вт
- Ср
- Чт
- Пт
- Сб
- Вс

Если пользователь создал лишний день, он отображается как:

- День 8
- День 9

Такие дни должны иметь предупреждающий визуальный статус.

## 5.3. Адаптив

### Desktop

- По возможности видно всю неделю сразу.
- Колонки дней узкие, но читаемые.
- Верхняя навигация остаётся компактной.
- Можно использовать hover-состояния: курсор вставки между событиями, подсветка замены при наведении на событие.

### Mobile

- Таблица недели скроллится горизонтально.
- Дни достаточно узкие, чтобы на вертикальном экране помещалось несколько дней.
- Hover-состояния не используются.
- Для редактирования включается отдельный режим “Редактирование”.
- В режиме просмотра кнопки вставки между событиями скрыты, чтобы не засорять экран.

## 5.4. Карточка активности в дне

Каждая карточка активности отображается так:

```text
07:00  Работа  11:00  4ч
```

Структура карточки:

- слева — время начала;
- по центру — название активности;
- справа — время окончания;
- ещё правее длительность (изменение сдвигает конец);
- фон карточки — цвет активности или группы;
- карточка имеет скруглённые края;
- карточка имеет лёгкую тень;
- при наведении или фокусе появляется мягкое выделение.

Карточки должны быть компактными. Рекомендуемый визуальный принцип:

- 1 час = примерно 20–28 px высоты;
- 5 минут = минимальный шаг данных;
- очень короткие активности могут иметь минимальную высоту для читаемости;
- фактическая длительность всё равно считается точно по минутам.

Высота карточки меняется при изменении временного промежутка.

### 5.4.1. Полоса текущего времени

На экране “Неделя” приложение показывает полосу текущего времени внутри сегодняшнего дня.

Правила отображения:

- полоса отображается только в текущем блоке активности или текущем свободном промежутке;
- положение полосы внутри блока считается как доля текущего времени между началом и концом именно этого блока/промежутка;
- это поведение важно сохранить: позицию полосы не нужно пересчитывать через глобальную проекцию всей недельной сетки;
- полоса занимает всю ширину текущего блока/промежутка, без внутренних боковых отступов;
- так как блоки скруглены и имеют `overflow: hidden`, полоса и её свечение должны корректно обрезаться у верхней и нижней границы блока, без вылезающих артефактов;
- визуальный центр полосы должен совпадать с рассчитанной временной позицией, без смещения вверх или вниз.

Визуальный стиль:

- в светлой теме полоса использует синие цвета, градиент и мягкое свечение;
- в тёмной теме полоса использует белые цвета, градиент и мягкое свечение;
- суммарная визуальная прозрачность должна быть около 50%;
- прозрачность задаётся только цветами (`rgba`, `hsla` и аналогичные форматы), без отдельного `opacity`;
- декоративное свечение должно быть собрано на самой полосе, например через многослойный `box-shadow`, без тяжёлых псевдоэлементов, которые создают грязную подложку.

Поведение во времени:

- положение полосы обновляется каждые 30 секунд, пока экран видим;
- это обновление не должно скроллить таблицу;
- при возврате в PWA/сайт после сворачивания или ухода со страницы приложение должно сразу синхронизировать текущее время;
- если пользователь отсутствовал 5 минут или больше, при возврате приложение может один раз доскроллить недельную таблицу к текущей полосе времени;
- логика доскролла после долгого отсутствия не должна быть завязана на 30-секундный интервал; она должна учитывать реальное время отсутствия и особенности PWA, где фоновые таймеры могут быть заморожены браузером или ОС.

## 5.5. Цвет карточки

Цвет активности становится фоном карточки.

Цвет текста должен оставаться читаемым при изменении фона.

Если в календарь добавлена группа, используется цвет группы.

Если в календарь добавлен лист-активность, используется цвет этой активности.

Текст на карточке должен автоматически подбираться по контрасту:

- на светлом фоне — тёмный текст;
- на тёмном фоне — светлый текст;
- в средних значениях — темный текст.

## 5.6. Добавление активности

Основной флоу:

1. Пользователь выбирает активность в панели выбора.
2. Наводит курсор между существующими событиями или кликает в пустое место дня.
3. Приложение показывает место вставки.
4. Пользователь кликает.
5. Активность добавляется с дефолтной длительностью.
6. Если нужно, пользователь меняет начало, конец или длительность.

На desktop:

- при наведении между событиями появляется тонкий курсор вставки;
- при наведении на существующую карточку видно, что выбранная активность может заменить текущую.

На mobile:

- пользователь включает режим “Редактирование”;
- между событиями появляются компактные кнопки добавления;
- при тапе по карточке открывается действие: заменить, редактировать, распилить, удалить.

## 5.7. Замена активности

Если выбрана активность и пользователь кликает по существующему блоку, происходит замена активности.

Важное правило:

**При замене сохраняется текущий временной промежуток.**

Пример:

- было: `Спорт`, 18:00–20:00;
- выбрали: `Чтение`, дефолтная длительность 30 минут;
- после замены: `Чтение`, 18:00–20:00.

Дефолтная длительность выбранной активности при замене не применяется.

## 5.8. Редактирование активности в дне

При клике по карточке открывается модальное окно или нижняя панель редактирования.

Поля:

- активность;
- день;
- время начала;
- время окончания;
- длительность;
- кнопка “Сохранить”;
- кнопка “Распилить”;
- кнопка “Удалить”.

Использовать выпадающее меню для отображения контролов активности.

Правила:

- шаг времени — 5 минут;
- при изменении начала, конца или длительности остальные значения пересчитываются;
- после сохранения блоки внутри дня сортируются по времени;
- наложения запрещены;
- если возникает наложение, приложение показывает ошибку и не сохраняет изменение;
- если появляется пустое место, оно остаётся пустым и отображается как зона для вставки.

## 5.9. Распил активности

Нужна возможность распилить событие на две части.

Флоу:

1. Пользователь открывает карточку активности.
2. Нажимает “Распилить”.
3. Видит текущую длительность.
4. Выбирает длительность первой и второй части.
5. Приложение создаёт две карточки подряд.
6. По умолчанию обе части имеют ту же активность.
7. После распила пользователь может заменить одну из частей другой активностью.

Пример:

- было: `Обучение`, 10:00–12:00;
- распил: первая часть 45 минут, вторая 75 минут;
- стало:
  - `Обучение`, 10:00–10:45;
  - `Обучение`, 10:45–12:00.

## 5.10. Пустые места

Пустое место между блоками должно быть видно, но не должно визуально перегружать экран.

Возможные варианты отображения:

- тонкая пунктирная зона;
- лёгкая серая область;
- на desktop — появляется только при hover;
- на mobile — появляется только в режиме редактирования.

Пустое место учитывается в статистике как “Не распределено”.

## 5.11. Конец дня в таблице

“Конец дня” отображается как тонкий системный блок или разделитель.

Пример:

```text
23:30  Конец дня
```

После него начинается следующая колонка.

Если блок активности пересекает “Конец дня”, приложение должно показать предупреждение. Для MVP можно не резать блок автоматически, но нужно явно подсветить, что структура дня неоднозначна.

## 5.12. Валидация недели

На экране недели должен быть блок предупреждений.

Проверки:

1. Количество логических дней меньше 7.
2. Количество логических дней больше 7.
3. Неделя не сходится в 168 часов.
4. Последний “Конец дня” не возвращает неделю к стартовому часу.
5. Есть пересечение активностей.
6. Есть активность, пересекающая системный “Конец дня”.
7. Есть активность без найденной активности в справочнике.

Важно: приложение предупреждает, но не удаляет данные автоматически.

# 6. Экран 2 — “Активности”

## 6.1. Назначение экрана

Экран нужен для создания справочника активностей, из которого пользователь потом “закупается” на экране недели.

Вложенность от 1 до 3, то есть на уровне 1 уже может быть лист.

Пользователь должен иметь возможность:

- создать группу;
- создать вложенную группу;
- создать активность-лист;
- задать название;
- задать дефолтную длительность;
- задать цвет;
- редактировать активность;
- удалить или архивировать активность;
- использовать как группу, так и лист в недельной таблице.

## 6.2. Иерархия

Максимальная вложенность — 3 уровня.

Пример:

```text
Спорт
  Зал
    Силовая тренировка
    Кардио
  Восстановление
    Растяжка
    Прогулка

Обучение
  Программирование
    Swift
    Архитектура
  Английский
    Speaking
    Vocabulary
```

Правила:

- на любом уровне может быть лист-активность если нет дочерних элементов;
- уровень 1 — группа;
- уровень 2 — подгруппа или активность;
- уровень 3 — лист-активность;
- любая группа тоже может быть добавлена в неделю как обычная активность;
- в календаре группа выглядит так же, как обычная активность;
- в статистике группа агрегирует вложенные активности.

## 6.3. Поля активности

Каждый элемент иерархии имеет поля:

- `id`;
- `parentId`;
- `name`;
- `color`;
- `defaultDurationMin`;
- `archived`;
- `createdAt`;
- `updatedAt`.

Даже группа имеет дефолтную длительность, потому что её можно добавить в неделю напрямую.

## 6.4. Дефолтная длительность

Дефолтная длительность применяется только при создании нового блока из справочника.

При замене существующего блока длительность сохраняется.

Шаг длительности — 5 минут.

Рекомендуемые быстрые значения:

- 5 минут;
- 10 минут;
- 15 минут;
- 30 минут;
- 45 минут;
- 1 час;
- 1 час 30 минут;
- 2 часа;
- 3 часа;
- 4 часа;
- 8 часов.

## 6.5. Цвета

Пользователь может выбрать цвет двумя способами:

1. Ввести HEX вручную.
2. Выбрать цвет из палитры.

Палитра:

- 8 основных цветовых направлений;
- у каждого направления 8 оттенков от светлого к тёмному;
- итого 64 цветных варианта;
- дополнительно 8 оттенков от белого к чёрному.

Итого: 72 быстрых цвета.

Цвета должны быть сдержанными, не кислотными. Продукт визуально должен оставаться строгим и “apple-like”, поэтому палитра должна быть приглушённой.

## 6.6. Удаление активности

Если активность не используется в неделе, её можно удалить.

Если активность уже используется в неделе, лучше не удалять её физически, а архивировать.

Поведение:

- архивная активность не показывается в выборе новых активностей;
- существующие блоки продолжают отображаться корректно;
- в карточке можно показать пометку “архив”.

# 7. Экран 3 — “Статистика”

## 7.1. Назначение экрана

Экран статистики показывает, как распределено время по активностям, группам, дням и всей неделе.

Главные вопросы:

- сколько часов ушло на каждую группу;
- сколько часов ушло на каждую конкретную активность;
- как распределение отличается по дням;
- сколько всего часов занято;
- сколько осталось нераспределённым;
- сходится ли неделя в 168 часов.

## 7.2. Вид статистики

Основной вид — таблица, похожая на иерархию активностей.

Колонки:

- Активность;
- Пн;
- Вт;
- Ср;
- Чт;
- Пт;
- Сб;
- Вс;
- Итого.

Если создан лишний день, добавляются колонки:

- День 8;
- День 9.

Такие колонки подсвечиваются как предупреждение.

## 7.3. Иерархическая агрегация

Статистика должна считать и группы, и листья.

Пример:

```text
Спорт                  6ч 00м
  Напрямую как Спорт   1ч 00м
  Зал                  3ч 00м
    Силовая            2ч 00м
    Кардио             1ч 00м
  Прогулка             2ч 00м
```

Важное правило:

Если пользователь добавил группу “Спорт” напрямую в неделю, это время учитывается в группе “Спорт”, но также должно быть видно отдельной строкой “Напрямую как Спорт”, чтобы не смешивать прямое использование группы с вложенными активностями.

## 7.4. Служебные строки

В статистике нужны служебные строки:

- Занято всего;
- Не распределено;
- Всего в дне;
- Отклонение от 24 часов;
- Всего в неделе;
- Отклонение от 168 часов.

“Конец дня” в статистике не считается.

## 7.5. Формат времени

Основной формат:

```text
2ч 30м
```

Дополнительно можно показывать десятичный формат в tooltip:

```text
2.5 часа
```

## 7.6. Цвета в статистике

Строки статистики используют те же цвета, что и активности, но очень деликатно:

- маленький цветной маркер слева;
- лёгкая цветная полоска;
- не заливать всю строку ярким цветом.

Статистика должна оставаться читаемой и строгой.

# 8. Настройки MVP

Минимальные настройки:

1. Пароль включён / выключен.
2. Начальное время недели.
3. Шаг времени: фиксированно 5 минут, в MVP можно не настраивать.
4. Масштаб высоты блоков.
5. Экспорт JSON.
6. Импорт JSON.
7. Сброс локального кэша PWA.

## 8.1. Авторизация

В MVP авторизация максимально простая:

- один пароль;
- без регистрации;
- без пользователей;
- без ролей;
- пароль задаётся на сервере;
- в интерфейсе можно отключить авторизацию и открыть сайт в публичный доступ.

Рекомендуемая модель:

- пароль хранится на сервере в переменной окружения `RYTHM_PASSWORD_HASH`;
- в состоянии приложения хранится настройка `authEnabled`;
- если `authEnabled = false`, сервер не требует пароль;
- если `authEnabled = true`, сервер требует авторизацию.

Так проще безопасно отключать и включать авторизацию из интерфейса, не меняя код.

## 8.2. Сессия

После ввода пароля сервер ставит cookie-сессию.

Требования:

- cookie httpOnly;
- срок жизни можно сделать 30 дней;
- logout очищает cookie;
- при отключённой авторизации cookie не нужна;
- сделать какой-то рефреш куки, чтобы при постоянных заходах сессия не протухала.

# 9. PWA и офлайн-режим

## 9.1. Цель PWA

Пользователь должен открыть приложение с телефона или ПК, даже если нет интернета, и спокойно редактировать таблицу.

После восстановления интернета приложение синхронизирует данные с сервером.

## 9.2. Что кэшируется

Service Worker кэширует:

- HTML;
- CSS;
- JS;
- локальные Bootstrap CSS и Bootstrap Icons;
- собранные Vite-ассеты Svelte-клиента;
- иконки PWA;
- manifest;
- app shell.

API-данные не должны полагаться только на HTTP cache. Состояние приложения хранится локально отдельно.

## 9.3. Локальное хранение

Рекомендуется использовать IndexedDB.

Для MVP можно сделать тонкую обёртку над IndexedDB или использовать небольшую локальную npm-зависимость, которая попадает в сборку и не грузится с CDN.

В локальном хранилище лежит последняя версия состояния приложения.

## 9.4. Алгоритм синхронизации

Принцип:

**более свежая версия побеждает.**

Мержей по отдельным дням и активностям в MVP нет.

Сравнение идёт по `updatedAt` всего состояния.

### При запуске приложения

1. Приложение читает локальное состояние из IndexedDB.
2. Если локальное состояние есть, сразу показывает его пользователю.
3. Параллельно пробует получить состояние с сервера.
4. Если сервер недоступен, пользователь продолжает работать офлайн.
5. Если сервер доступен, приложение сравнивает `updatedAt`.
6. Если сервер свежее, локальное состояние заменяется серверным.
7. Если локальное состояние свежее, оно полностью отправляется на сервер.
8. Если версии равны, ничего не происходит.

### При редактировании офлайн

1. Пользователь меняет неделю или активности.
2. Приложение обновляет локальное состояние.
3. Приложение ставит новый `updatedAt`.
4. Приложение помечает состояние как `dirty`.
5. При появлении интернета состояние отправляется на сервер целиком.

### При конфликте

Конфликт не мержится.

Правило простое:

- если серверный `updatedAt` больше — побеждает сервер;
- если локальный `updatedAt` больше — побеждает клиент;
- если равны — ничего не делаем.

Если более свежая версия перетёрла другую, это считается допустимым поведением MVP.

## 9.5. Индикатор синхронизации

В интерфейсе нужен компактный статус:

- Сохранено;
- Офлайн;
- Есть локальные изменения;
- Синхронизация;
- Ошибка синхронизации;
- Серверная версия свежее;
- Локальная версия отправлена на сервер.

# 10. Хранение данных

## 10.1. Рекомендуемый вариант MVP

Для MVP лучше использовать JSON-файл.

Причины:

- один пользователь;
- один сценарий недели;
- нет сложных запросов;
- легко бэкапить;
- легко открыть руками;
- легко перенести сервер;
- не нужно поднимать отдельную базу.

Рекомендуемый путь:

```text
/app/data/state.json
```

В docker compose этот путь должен быть volume, чтобы данные не пропадали при пересборке контейнера.

## 10.2. Атомарная запись

Чтобы не повредить JSON при падении сервера, запись должна быть атомарной:

1. Сервер пишет новое состояние в `state.json.tmp`.
2. После успешной записи делает rename в `state.json`.
3. Старый файл заменяется новым атомарно.

## 10.3. Бэкапы

Перед заменой состояния можно хранить последние N копий.

Рекомендуется для MVP:

- хранить последние 20 снимков;
- путь: `/app/data/backups`;
- имя файла: `state-2026-05-20T19-30-00-000Z.json`.

Это не усложняет продукт, но спасает при случайной перезаписи.

## 10.4. Когда понадобится SQLite

SQLite можно рассмотреть позже, если появятся:

- несколько пользователей;
- история изменений;
- несколько шаблонов;
- аналитика за разные периоды;
- частичная синхронизация;
- нормальные миграции данных.

Для MVP JSON проще и лучше.

## 11.1. Почему absolute minutes

`startAbsMin` и `endAbsMin` считаются от начала недели.

Если неделя начинается в понедельник в 07:00, то:

- `0` = Пн 07:00;
- `60` = Пн 08:00;
- `1020` = Пн 00:00 следующего календарного дня;
- `1440` = через 24 часа после старта недели;
- `10080` = через 168 часов после старта недели.

Это удобно, потому что:

- день может залезать за полночь;
- можно проверять сумму 168 часов;
- можно строить логические дни через маркеры `dayEnd`;
- можно разрешить ошибочные 8 дней без удаления данных;
- статистику считать проще.

## 11.2. Отображение времени

Для отображения времени используется формула:

```text
displayClockMin = (weekStartClockMin + startAbsMin) % 1440
```

Пример:

- `weekStartClockMin = 420`, то есть 07:00;
- `startAbsMin = 60`;
- отображаемое время = 08:00.

# 12. API MVP

## 12.1. GET /api/state

Возвращает текущее состояние.

Ответ:

```json
{
  "ok": true,
  "state": {
    "schemaVersion": 1,
    "updatedAt": "2026-05-20T19:30:00.000Z",
    "settings": {
      "authEnabled": true,
      "weekStartClockMin": 420,
      "timeStepMin": 5,
      "pxPer5Min": 2,
      "firstDayLabel": "Пн",
      "theme": "system"
    },
    "activities": [],
    "timeline": []
  }
}
```

## 12.2. PUT /api/state

Полностью заменяет состояние на сервере, если клиентская версия свежее или если сервер принимает forced overwrite.

Тело запроса:

```json
{
  "state": {
    "schemaVersion": 1,
    "updatedAt": "2026-05-20T19:35:00.000Z",
    "settings": {
      "authEnabled": true,
      "weekStartClockMin": 420,
      "timeStepMin": 5,
      "pxPer5Min": 2,
      "firstDayLabel": "Пн",
      "theme": "system"
    },
    "activities": [],
    "timeline": []
  }
}
```

Ответ:

```json
{
  "ok": true,
  "serverUpdatedAt": "2026-05-20T19:35:00.000Z"
}
```

Если серверная версия свежее:

```json
{
  "ok": false,
  "code": "SERVER_STATE_IS_NEWER",
  "serverUpdatedAt": "2026-05-20T19:40:00.000Z"
}
```

## 12.3. GET /api/auth/status

Возвращает состояние авторизации.

Ответ:

```json
{
  "ok": true,
  "authEnabled": true,
  "authenticated": true
}
```

Если авторизация отключена, `authenticated` возвращается как `true`.

## 12.4. POST /api/auth/login

Проверяет пароль и ставит cookie.

Тело:

```json
{
  "password": "123456"
}
```

Ответ:

```json
{
  "ok": true
}
```

Пароль `123456` в примере нужен только как демонстрация формы запроса. В реальном деплое пароль задаётся владельцем сервера.

## 12.5. POST /api/auth/logout

Очищает cookie.

Ответ:

```json
{
  "ok": true
}
```

## 12.6. GET /api/health

Для проверки контейнера и nginx.

Ответ:

```json
{
  "ok": true,
  "app": "rythm"
}
```

# 15. Docker и деплой

## 15.1. Dockerfile

Сервер должен собираться в Docker-образе.

Требования:

- Node.js runtime;
- установка зависимостей через npm;
- сборка frontend;
- запуск production-сервера;
- порт внутри контейнера: 3000.

## 15.2. docker-compose.yml

Требования:

- сервис `rythm`;
- проброс порта `${PORT:-3000}:3000`;
- volume для `/app/data`;
- restart policy `unless-stopped`;
- env-переменные для пароля и cookie secret.

Пример логики compose:

```yaml
services:
  rythm:
    build: .
    container_name: rythm
    restart: unless-stopped
    ports:
      - "${PORT:-3000}:3000"
    volumes:
      - ./data:/app/data
    environment:
      NODE_ENV: production
      PORT: 3000
      RYTHM_PASSWORD_HASH: "${RYTHM_PASSWORD_HASH:-}"
      RYTHM_COOKIE_SECRET: "${RYTHM_COOKIE_SECRET:-rythm-local-secret-change-after-first-deploy}"
```

Хеш пароля генерируется командой `npm run hash-password -- "my-password"` или интерактивной настройкой `npm run setup:env`, которая записывает `.env` для `docker compose`.

## 15.3. Команды деплоя

Первичный запуск на VPS внутри папки проекта:

```bash
docker compose up -d --build
```

Локальный деплой на VPS выполняется скриптом:

```bash
npm run deploy:vps
```

По умолчанию скрипт использует SSH host `frity.vds`, директорию `/var/www/rythm.frylo.org` и ветку `main`. Параметры переопределяются переменными окружения или файлом `.env.deploy`:

```bash
RYTHM_DEPLOY_HOST=my-vps
RYTHM_DEPLOY_USER=root
RYTHM_DEPLOY_PASSWORD=optional-ssh-password
RYTHM_DEPLOY_DIR=/opt/rythm
RYTHM_DEPLOY_BRANCH=main
```

На VPS Node.js/npm не нужны: `npm ci` и `npm run build` выполняются внутри Docker-образа.

## 15.4. Nginx

Nginx должен проксировать домен на локальный порт 3000.

Продукт не обязан сам заниматься SSL. SSL можно настроить через nginx и certbot.

# 16. Визуальный стиль

## 16.1. Общий стиль

Стиль строгий, сдержанный, apple-like.

Основные ощущения:

- чистота;
- воздух;
- аккуратные карточки;
- мягкие тени;
- сдержанные градиенты;
- минимум визуального шума;
- интерфейс похож на инструмент, а не на игрушку.

## 16.2. Цветовой принцип 60-30-5-5

Рекомендуемая пропорция:

- 60% — белые и почти белые поверхности;
- 30% — чёрный и тёмно-серый текст;
- 5% — серые градиенты, разделители, панели;
- 5% — цвета активностей.

Цвета активностей должны помогать различать смысловые блоки, но не превращать интерфейс в яркую мозаику.

Принцип выше также инвертировать в темные тона. Делаем сразу светлую и темную тему, с автоматическим переключением "как в системе" и ручным.

## 16.3. Типографика

Использовать системные шрифты:

```css
font-family: -apple-system, BlinkMacSystemFont, "Segoe UI", system-ui, sans-serif;
```

Принципы:

- компактные подписи времени;
- средний вес для названий активностей;
- не использовать декоративные шрифты;
- высокая читаемость на мобильном экране.

## 16.4. Скругления

Рекомендуемые радиусы:

- карточки активностей: 10–14 px;
- панели: 16–24 px;
- кнопки: 10–14 px;
- модальные окна: 20–28 px.

## 16.5. Тени

Тени должны быть мягкими и почти незаметными.

Пример направления:

```css
box-shadow: 0 8px 24px rgba(15, 23, 42, 0.08);
```

## 16.6. Микроинтеракции

Нужны лёгкие анимации:

- hover карточки;
- появление курсора вставки;
- открытие модального окна;
- переключение режима редактирования;
- сохранение;
- ошибка валидации;
- изменение статуса синхронизации.

Рекомендуемая длительность:

- 120–180 ms для быстрых действий;
- 200–260 ms для модальных окон.

Анимации должны быть спокойными, без “прыгающего” интерфейса.

# 17. Основные пользовательские сценарии

## 17.1. Создание активности

1. Пользователь открывает экран “Активности”.
2. Нажимает “Добавить”.
3. Вводит название “Английский”.
4. Выбирает родителя “Обучение”.
5. Ставит дефолтную длительность 45 минут.
6. Выбирает цвет.
7. Сохраняет.
8. Активность появляется в иерархии и в панели выбора на экране недели.

## 17.2. Добавление активности в день

1. Пользователь открывает экран “Неделя”.
2. Выбирает активность “Английский”.
3. Наводит курсор между событиями во вторник.
4. Видит курсор вставки.
5. Кликает.
6. Создаётся блок с длительностью 45 минут.
7. Блок автоматически получает цвет активности.

## 17.3. Замена активности

1. Пользователь выбирает активность “Прогулка”.
2. Кликает по блоку “Рилсы”, который длится 1 час.
3. Блок становится “Прогулка”.
4. Время 21:00–22:00 сохраняется.

## 17.4. Распил блока

1. Пользователь открывает блок “Отдых” на 2 часа.
2. Нажимает “Распилить”.
3. Делит на 30 минут и 1 час 30 минут.
4. Первую часть заменяет на “Рилсы”.
5. Вторую часть заменяет на “Чтение”.

## 17.5. Завершение дня

1. Пользователь добавляет системную активность “Конец дня”.
2. Всё, что находится после неё, попадает в следующий день.
3. Если таких маркеров стало 8, приложение показывает предупреждение.

## 17.6. Работа офлайн

1. Пользователь открывает PWA в дороге без интернета.
2. Таблица загружается из локального хранилища.
3. Пользователь меняет расписание.
4. Приложение показывает статус “Офлайн, есть локальные изменения”.
5. Интернет появляется.
6. Приложение отправляет всю свежую версию на сервер.
7. Статус меняется на “Сохранено”.

# 18. Ограничения MVP

В MVP не делаем:

- несколько шаблонов жизни;
- историю разных периодов;
- регистрацию пользователей;
- роли;
- совместный доступ;
- Google Calendar integration;
- уведомления;
- drag-and-drop resize;
- drag-and-drop reorder как обязательную механику;
- сложный конфликт-мердж;
- серверную базу PostgreSQL/MySQL;
- аналитику по месяцам;
- публичную галерею шаблонов;
- мобильное native-приложение.

Это можно рассмотреть после проверки MVP.

# 19. Критерии готовности MVP

## 19.1. Product acceptance

MVP считается готовым, если пользователь может:

- открыть сайт;
- ввести пароль;
- увидеть недельную таблицу;
- создать активность;
- создать вложенность до 3 уровней;
- выбрать цвет активности;
- добавить активность в день;
- заменить активность;
- изменить начало, конец и длительность;
- распилить блок на 2 части;
- добавить “Конец дня”;
- увидеть предупреждение при неправильной структуре недели;
- посмотреть статистику по дням и неделе;
- открыть приложение офлайн;
- изменить данные офлайн;
- синхронизировать данные после появления интернета.

## 19.2. Design acceptance

Дизайн считается готовым, если:

- вся неделя читается на desktop;
- на mobile таблица удобно скроллится горизонтально;
- несколько дней помещаются на вертикальном мобильном экране;
- карточки компактные;
- цвета активностей не ломают общий строгий стиль;
- режим просмотра не перегружен кнопками;
- режим редактирования явно отличается от режима просмотра;
- интерфейс выглядит сдержанно, современно и аккуратно.

## 19.3. Development acceptance

Разработка считается готовой, если:

- проект запускается через npm;
- проект собирается без CDN;
- Svelte-клиент собирается Vite в `public/assets`;
- Bootstrap и Bootstrap Icons лежат локально в `public/vendor`;
- PWA manifest подключён;
- Service Worker кэширует app shell;
- состояние хранится в IndexedDB;
- сервер хранит состояние в JSON-файле;
- запись JSON атомарная;
- docker compose поднимает приложение;
- volume сохраняет данные;
- контейнер перезапускается через `restart: unless-stopped`;
- `/api/health` отвечает;
- nginx может проксировать приложение на порт 3000.
